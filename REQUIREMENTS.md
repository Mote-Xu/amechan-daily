## P2：公共部署 🔴 当前重点

**目标**：将项目部署到公网，同时保护 API 不被滥用。

### 安全需求

| ID | 需求 | 说明 |
|----|------|------|
| S1 | API 鉴权 | 生成 API 需要 token，不能裸奔。只消耗自己的 DeepSeek 额度 |
| S2 | 频率限制 | 防止单 IP 短时间内大量调用，刷爆 API |
| S3 | CORS 白名单 | 只允许指定域名访问 POST 端点 |
| S4 | Token 刷新 | Token 有过期机制，前端自动刷新，不能写死在代码里 |

### 部署需求

| ID | 需求 | 说明 |
|----|------|------|
| D1 | HTTPS | 必须 https，不能用 http 明文 |
| D2 | 静态资源 | HTML/CSS/JS/图片 部署到 CDN 或静态托管 |
| D3 | 后端 | Python 服务需要托管（云服务器/Serverless） |
| D4 | 低成本 | 个人项目，成本敏感 |

### 需要 Gemini 帮助的问题

1. **架构方案**：推荐的部署架构？前端放 GitHub Pages/Vercel？后端放哪里（AutoDL/阿里云/Cloudflare Workers/其他）？
2. **API 保护**：token 机制的最佳实践？CSRF？频率限制方案？
3. **DeepSeek API 中转**：是否需要在后端加一层缓存/队列，防止前端请求直接打到 DeepSeek？
4. **其他风险**：部署到公网还有哪些需要考虑的安全问题？

### 约束
- conda `deepseek_v4_api`，Python 3.11，DeepSeek V4-pro
- 纯 HTML/CSS/JS，零框架
- 低成本，个人项目
